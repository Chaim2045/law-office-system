## 🧪 5 טרקים של חקירה — כל אחד סאב-סוכן נפרד

הפעל את 5 הטרקים הבאים **במקביל** דרך Task tool:

### TRACK A — "Navigator Reality Test"
**השערה לאמת:** הסוכן `navigator-process-guide` אינו יכול לייצר דיווח אמין על מצב העבודה, כי אין לו כלים לאמת git state.

**סאב-סוכן — מה עליו לעשות:**
1. לקרוא את `.claude/agents/navigator.md` ולזהות את רשימת ה-tools.
2. לייצר 3 תרחישים אמיתיים של "אני לא זוכר איפה הייתי":
   - תרחיש 1: יש branch feature/X פתוח, יש commit לא-pushed, יש PR פתוח.
   - תרחיש 2: יש working directory clean, PROD-stable 12 commits מאחורי main.
   - תרחיש 3: יש uncommitted changes בסוכנים.
3. לקבוע מה התשובה ה**נכונה** לכל תרחיש (דרך הרצת git status / gh pr list).
4. להפעיל את navigator דרך **Task tool עם `subagent_type: "navigator-process-guide"`** (השם המדויק לפי frontmatter של `.claude/agents/navigator.md`). תן לו prompt שמתאר את התרחיש + "איפה אני בפרוטוקול?".
5. להשוות: האם navigator ענה נכון? באיזה אחוז? האם הוא המציא שדות?

**פלט נדרש:** טבלה של 3 תרחישים × תשובה-נכונה × תשובת-navigator × דיוק.

### TRACK B — "Hooks Feasibility Study"
**השערה לאמת:** `.claude/settings.json` של Claude Code תומך ב-UserPromptSubmit hooks באופן שמאפשר אכיפה של work-session-gatekeeper.

**סאב-סוכן — מה עליו לעשות:**
1. לקרוא את `.claude/settings.json` ו-`.claude/settings.local.json` הנוכחיים.
2. לקרוא את תיעוד Claude Code על hooks (דרך WebFetch מ-docs.claude.com, או מ-CHANGELOG מקומי אם יש).
3. לזהות: מה ה-syntax המדויק של `UserPromptSubmit`? אילו matcher supported? האם output של hook מוזרק ל-context?
4. לכתוב דוגמה קונקרטית של hook שיריץ `git status + gh pr list` וידביק למסך.
5. לבדוק — האם יש hooks existing בפרויקטים אחרים ב-`/sessions/vigilant-quirky-ptolemy/mnt/` שאפשר ללמוד מהם?
6. להעריך סיכונים: latency, stdout pollution, permission prompts.

**פלט נדרש:** feasibility report: PASS / PARTIAL / BLOCKED. אם PASS — דוגמת JSON מוכנה. אם PARTIAL — מה חסר.

### TRACK C — "Agent↔Command Drift Proof"
**השערה לאמת:** 3 הגרסאות של "code review" (reviewer.md / review-strict.md / ביקורת.md) מייצרות פלטים שונים מהותית באותו תרחיש.

**סאב-סוכן — מה עליו לעשות:**
1. לבחור קטע קוד אמיתי מהפרויקט שיש בו ליקוי בינוני (למשל: פונקציה ב-`apps/user-app/js/` שבאמת קיימת).
2. להפעיל את **שלוש הגרסאות** על אותו קטע קוד, כל אחת בקריאת Task נפרדת:
   - **גרסה 1 — דרך Task tool** עם `subagent_type: "code-reviewer"` (הסוכן עצמו). prompt: "בצע code review על הקובץ X".
   - **גרסה 2 — דרך Task tool** עם `subagent_type` שמתאים לפקודה `/review-strict` (בדוק איזה סוכן היא מפעילה — קרא את `.claude/commands/review-strict.md`). 
   - **גרסה 3 — דרך Task tool** עם ה-subagent שמתאים ל-`/ביקורת` (קרא את `.claude/commands/ביקורת.md`).
3. לנתח: האם הממצאים זהים? שונים? סותרים? איזה עמוק יותר? איזה פספס?
4. לעשות את אותו דבר עבור **גם** שלוש גרסאות של PROD gate: `prod-gatekeeper` agent, `/validate-strict`, `/ולידציה`.

**פלט נדרש:** טבלת השוואה + מסקנה: האם ה-drift אמיתי ומזיק, או שזה רק סגנון?

### TRACK D — "The Null Test — Agent by Agent"
**השערה לאמת:** מחלק מ-17 הסוכנים ניתן למחוק בלי אובדן תפקוד.

**סאב-סוכן — מה עליו לעשות:**
1. לייצר טבלה של כל 17 הסוכנים.
2. לכל סוכן, לזהות:
   - מה הוא עושה ייחודית (שאף אחד אחר לא).
   - איזה סוכן אחר יכול לכסות 80%+ מתפקידו.
   - תרחיש ברור שבו **אי-אפשר** להחליף אותו (אם יש).
3. לדרג כל סוכן כ: **חיוני / מיוחד-לשוליים / מותרי / מיותר**.
4. להריץ Null Test עבור 3 סוכנים "מיותרים" מובהקים — להפעיל סוכן אחר שלכאורה יחליף, ולבדוק אם הפלט מספק.

**פלט נדרש:** טבלת 17 סוכנים × דירוג × הצדקה × ממצא Null Test.

### TRACK E — "The 5-Step Fatigue Test"
**השערה לאמת:** החלפת 5 שלבי "דבר בלי לראות" ב-4 שלבים מעלה את איכות הפלט הסופי.

**סאב-סוכן — מה עליו לעשות:**
1. לבחור משימה אמיתית טריוויאלית מהפרויקט (למשל: "תקן typo ב-CSS").
2. לבצע את המשימה דרך הפרוטוקול המלא של 5 שלבים: intent-refiner → /טומי → /אבחון → /תכנון → code.
3. לתעד: כמה סיבובי שיחה? כמה טוקנים? מתי הסוכן הראשון הרגיש "חייב לספק"? האם היה ממצא קפוץ?
4. לבצע את אותה משימה דרך פרוטוקול מקוצר: intent-refiner → /תכנון → code. (שני שלבים פחות).
5. להשוות: איכות תוצר, מהירות, איכות תקשורת עם חיים.

**פלט נדרש:** report של 2 ריצות, עם ציון איכות לכל אחת.
